# SystemSculpt 5.3.1

## What's New

SystemSculpt 5.3.1 is a native release-hardening patch for the Pi-backed plugin workflow. The headline change is that release verification now runs through one real macOS, Windows, and Android matrix, and the provider-connected Pi lane uses the supported OpenRouter GPT-5.4 Mini path instead of stale direct-OpenAI assumptions.

## Highlights

- Added a first-class native release matrix that gates macOS desktop baselines, Windows clean-install parity, Windows desktop baselines, Android sync and headless relaunch, and Android runtime smoke before draft creation.
- Hardened the Windows no-focus automation lane so transient hosted send failures are classified separately from real plugin regressions, with combined baseline summaries and explicit managed recovery checks.
- Defaulted provider-connected desktop and Windows parity checks to OpenRouter using `openai/gpt-5.4-mini`, which matches the supported Pi-connected provider path in this environment.
- Tightened Pi auth storage, vault-path resolution, Studio provider inventory, and desktop automation bridge behavior so Settings, Studio, and native automation all read from the same bundled auth and storage model.
- Scrubbed machine-specific tracked paths from release-facing fixtures, examples, and remote temp defaults so the release safety preflight now passes cleanly before tag or draft creation.

## Important Notes

- This is a patch release focused on reliability, validation, and release safety. It does not introduce a new end-user product surface.
- Provider-connected release validation is intentionally OpenRouter-first for the Pi lane in this environment. Direct OpenAI provider credits are not part of the supported release path here.
- iOS runtime smoke was intentionally skipped in this release pass because no physical iOS device was connected. macOS, Windows, and Android all passed.
- No manual migration should be required for existing users beyond the normal plugin update.
