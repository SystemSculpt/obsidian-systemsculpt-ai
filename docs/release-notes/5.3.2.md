# SystemSculpt 5.3.2

## What's New

SystemSculpt 5.3.2 is a reliability and UX patch for the Pi-backed plugin workflow. The headline change is that local Pi tool calls now stay on the local Pi path end to end instead of being mistaken for hosted follow-up work, while a separate round of user-reported UX regressions around transcription, ribbons, Studio open, and chat visibility is also cleaned up in the same release line.

## Highlights

- Local Pi tool executions now surface in ChatView as completed local tool calls, with their final result attached, instead of being reinterpreted as hosted external tool requests on the next turn.
- The provider-connected native desktop baseline now exercises a real filesystem tool turn on `local-pi-openrouter@@openai/gpt-5.4-mini` on both macOS and Windows, including completed tool-call assertions rather than only checking response text.
- Studio provider inventory now reads the canonical bundled Pi auth storage directly. The old runtime-relative fallback that could silently hide broken production reads in the built plugin is gone.
- Windows release prep now auto-refreshes stale plugin runtimes through the canonical remote bootstrap relaunch path and redacts sensitive bootstrap report fields before printing them.
- Meeting processor outputs now stay associated with renamed audio files, so a processed note does not quietly fall back to an unprocessed state after the source recording is renamed.
- The transcription clean-up prompt setting is restored, mobile recording now prefers `m4a` when the platform supports it, chat can hide system-role messages again, Studio can auto-create a project on first open, and the ribbon lane is less intrusive.

## Important Notes

- This is a patch release focused on reliability, release confidence, and UX polish. It does not add a new top-level product surface.
- Provider-connected verification for the Pi lane remains OpenRouter-first in this environment, using `openai/gpt-5.4-mini` as the supported release-check target.
- The auth-inventory change is intentionally strict: if the canonical bundled Pi auth path breaks again, the failure should surface instead of being hidden behind a silent fallback.
- `npm run check:release:native` passed on macOS, Windows, and Android during this release pass. iOS runtime smoke remains optional and unavailable in this environment because no physical iOS device was attached.
- No manual migration should be required for existing users beyond the normal plugin update.

## API and Platform Work Included

- Local Pi streaming now preserves completed tool-call state and tool results from `tool_execution_end` instead of inferring completion from a later stop reason.
- ChatView streaming and continuation logic now respects already-completed local Pi tool calls instead of trying to continue them through the hosted tool loop.
- Native desktop automation baselines now assert real provider-connected filesystem tool turns and completed tool-call counts on macOS and Windows.
- Windows clean-install and desktop-automation runners now compare the live bridge plugin version against the repo's `manifest.json` and automatically run the canonical remote bootstrap relaunch when the runtime is stale.
- Windows bootstrap output now redacts license, identity, and token-like fields before printing, and provider-connected automation classifies the Win32 console-title pipe error as a transient candidate retry instead of a deterministic plugin failure.
- Studio provider auth inventory now uses the bundled Pi auth storage directly, so release builds do not silently diverge from the canonical runtime path.
- UX fixes in this release line include meeting-processor rename tracking, restored transcription prompt controls, mobile recorder format preference cleanup, less aggressive ribbon behavior, first-open Studio project creation, and restored system-message visibility controls in chat.
